# ADR-002 — Multi-tenancy & Celery queue model

- Status: **Accepted**
- Date: 2026-06-08
- Context: [high-level-architecture.md](./high-level-architecture.md), overview §5

## Context

SaaS, каждый пользователь полностью изолирован, но источники (каналы) дороги и пересекаются между юзерами. Нужно: (а) изоляция данных по тенанту, (б) отсутствие параллельного запуска батча одного юзера, (в) единичное чтение общего канала.

## Decision

1. **Изоляция данных по `user_id`.** Все пользовательские таблицы (`watchlists`, `clusters`, `scores`, `alerts`) несут `user_id` (FK→`users`), все запросы фильтруются по нему. Репозитории принимают `user_id` обязательным параметром — нет «глобальных» выборок пользовательских данных.

2. **Per-user очереди.** Celery-очередь именуется `batch:user_{id}`; задача батча юзера ставится с `max_instances=1`-семантикой (новый батч не стартует, пока предыдущий не закончил) — через единый per-user lock (Redis) + идемпотентный drain буфера. Beat enqueue'ит батчи для активных юзеров раз в минуту; scorer — отдельный тик раз в 5 минут.

3. **Cross-tenant дедупликация источников.** `channels` (источники) — глобальная таблица; связь с юзерами через `watchlists` (junction). Коллектор читает **объединённое** множество уникальных `SourceRef` по всем активным watchlist'ам — канал читается один раз, сырые посты кладутся в общий Redis-буфер ключом по источнику; батч юзера фильтрует буфер по его watchlist'у. Это даёт экономию rate-limit пула (overview §2/§5).

4. **Retention.** Сырой контент постов — TTL ≤ 48h (Redis/temp), в Postgres хранятся метрики + векторы + кластеры. Удаление аккаунта вычищает все строки с его `user_id` одним каскадом.

## Consequences

- (+) Линейное масштабирование чтения: N юзеров на один канал = одно чтение.
- (+) Нет гонок батча одного юзера; изоляция тенантов на уровне схемы и репозиториев.
- (+) Дешёвый rate-limit бюджет → меньше технических аккаунтов в пуле.
- (−) Сложнее модель буфера (общий по источнику + фильтр по watchlist на батче).
- (−) Нужен надёжный per-user lock (Redis) и идемпотентный drain.
- Влияет на задачи: **task-002** (schema + FK + индексы по `user_id`), **task-006** (Celery очереди/locks/beat), **task-005** (общий буфер по источнику), **task-011** (retention/GDPR-delete).
