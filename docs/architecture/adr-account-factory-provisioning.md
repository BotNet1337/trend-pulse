# ADR — account-factory: провижининг TG-аккаунтов + activation gating

- Status: **Accepted**
- Date: 2026-06-20
- Context: TASK-132..137, [adr-005-infra-provisioning-and-secrets.md](./adr-005-infra-provisioning-and-secrets.md)

## Context

Пул технических Telegram-аккаунтов — критическая инфраструктура сборщика: без
аккаунтов нет вступлений в каналы, нет инжеста. Ручное пополнение пула не масштабируется.
Нужен автоматический провижинер (Layer B), который:
- периодически проверяет уровень пула и докупает аккаунты через SMS-провайдер (SMSPVA);
- не должен требовать изменений в Celery-роутинге или отдельного брокера;
- должен быть **безопасен по умолчанию** — никаких реальных покупок без явной активации;
- должен масштабироваться независимо от основного worker'а (embed-батчи и factory-тики
  имеют разные профили нагрузки и ресурсов).

## Decision

### 1. Отдельный контейнер `account-factory` (Celery worker)

Выделенный Celery-worker, потребляющий очередь `celery` по умолчанию (`-Q celery`).
`factory_tick` — BEAT-задача (`factory.tasks.factory_tick`, константа `FACTORY_TICK_TASK`),
зарегистрированная в `backend/src/scheduler.py` как `"factory-tick"`. Задача **unrouted**
(без явного `queue=`), поэтому приземляется на очередь `celery` — ту же, что потребляют
основной worker и новый `account-factory`.

Celery доставляет каждое сообщение **ровно одному** потребителю очереди — дублирования нет.
Это означает, что `factory_tick` в каждый конкретный момент обрабатывает либо `worker`,
либо `account-factory`, но не оба. Выбор потребителя недетерминирован (round-robin/prefetch),
поэтому выделенный контейнер даёт изоляцию вероятностно, а не гарантированно (см. (−) ниже).

Преимущества выделенного контейнера:
- **Независимое масштабирование**: factory-нагрузка (часовой тик, SMSPVA HTTP, MTProto
  регистрация) не конкурирует с embed-батчами за CPU/память.
- **Изоляция отказов**: зависание factory (SMSPVA timeout) не блокирует scoring-pipeline.
- **Ресурсные лимиты**: factory — 1 CPU / 1024 МБ (prod swarm); worker — 2 CPU / 3072 МБ.
- **Нет изменений в backend-роутинге**: задача уже unrouted, новый worker её подхватит
  без правок в `scheduler.py` или `celery_app`.

### 2. Provider-driven activation gating (не boolean-флаг)

Активация через значение переменной `ACCOUNT_FACTORY_PROVIDER`:

| Значение        | Поведение                                                              |
|-----------------|------------------------------------------------------------------------|
| `""` / не задан | **No-op tick** — мгновенный возврат, 0 сетевых вызовов. Дефолт в prod. |
| `fake`          | **Dev-режим** — полная логика без сетевых запросов (network-free CI). Дефолт в dev. |
| `smspva`        | **Реальный провижининг** через SMSPVA API.                             |

Решение «провайдер как гейт» (а не отдельный `ENABLED=true/false`) выбрано потому, что:
- нет смысла указывать провайдера без его активации — они неразделимы;
- пустая строка — интуитивно понятный «выключатель» без лишнего флага;
- `fake` даёт CI-покрытие без живых учётных данных.

### 3. USD hard-cap + испытательный срок

- `ACCOUNT_FACTORY_BUDGET_USD` — кумулятивный лимит расходов. `"0.00"` = hard-cap,
  отклоняющий любую покупку даже при включённом провайдере. Dev-дефолт намеренно `"0.00"`.
- `ACCOUNT_FACTORY_PROBATION_DAYS` — период warm-up нового аккаунта (дефолт 14 дней)
  перед попаданием в live-пул. Защищает пул от сразу большого притока необкатанных сессий.

### 4. SMSPVA_API_KEY — только через ansible-vault

Ключ SMSPVA никогда не коммитится в открытом виде. Хранится в
`ops/ansible/vault/sensitive.vault.yml` (`vault_smspva_api_key`), рендерится в
`sensitive.env` через шаблон `ops/ansible/roles/env/templates/sensitive.env.j2`:

```jinja2
SMSPVA_API_KEY={{ vault_smspva_api_key | default('') }}
```

`default('')` — чтобы `make ansible-unpack` не падал на хостах, где vault-ключ ещё не
добавлен. Пустое значение заставляет `smspva`-провайдер упасть при первом тике с ясным
сообщением об ошибке (fail fast > silent no-op при неверной конфигурации).

**Rotate-after-exposure**: если ключ скомпрометирован — немедленно сгенерировать новый
в личном кабинете SMSPVA, обновить vault, передеплоить. Старый ключ деактивировать.

### 5. Egress — только для `smspva`-провайдера

Контейнер `account-factory` подключён к сети `egress` (non-internal) для:
- SMSPVA HTTPS (аренда номера телефона);
- Telegram MTProto (регистрация нового аккаунта).

В dev с `fake`-провайдером egress не используется, но наличие сети не создаёт рисков —
сеть non-internal только для `account-factory` и `worker`, которые и раньше имели egress.

## Consequences

- (+) Пул TG-аккаунтов растёт автоматически при активации — без ручных вмешательств.
- (+) Безопасен по умолчанию: пустой провайдер в prod = 0 расходов, 0 API-вызовов.
- (+) Независимо масштабируется (отдельный контейнер, отдельные ресурсные лимиты).
- (+) Нет изменений в Celery-роутинге — factory_tick подхватывается без правок backend.
- (+) `fake`-провайдер даёт CI-покрытие без реальных учётных данных.
- (−) factory_tick может обработать основной `worker` (если стартует раньше account-factory);
  это не баг — задача идемпотентна и не критична по времени (hourly tick).
- (−) Для активации требуются три явных шага (prod.yml + vault + deploy) — намеренная
  защита от случайной активации.
