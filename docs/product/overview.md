# TrendPulse — Product Document

> Персональный детектор вирусного контента из Telegram

**Статус:** MVP построен (эпики A–D done), pre-revenue — текущая волна **Epic E «Path to revenue»** ([epics](./epics/README.md))  
**Стек:** Python · Celery + Redis · PostgreSQL + pgvector · FastAPI · React (frontend + landing)  
**Модель:** Multi-tenant SaaS  

---

## 1. Что это и зачем

TrendPulse — сервис который мониторит публичные Telegram-каналы и находит контент, который резко набирает распространение — раньше, чем тема стала мейнстримом.

Пользователь указывает список публичных каналов и тему интереса (крипто, футбол, политика, и т.д.). Система в реальном времени отслеживает эти каналы через пул технических аккаунтов, кластеризует похожие новости across каналов, считает viral score — и присылает сигнал.

**Итоговый результат для пользователя:**

```
🔥 Viral alert [crypto] — "Bitcoin ETF approval"
Score: 94 · 47 каналов за 23 мин · first seen 14:02
```

## 2. Как работает мониторинг (архитектурное решение)

### Почему не userbot пользователя

Telegram не предоставляет OAuth с ограниченными правами. `session_string` — это полный доступ к аккаунту (сообщения, контакты, возможность писать от имени). Хранить чужую сессию на сервере — юридическая проблема и нарушение ToS Telegram §5.2.

### Правильный подход — пул технических аккаунтов

```
Пользователь указывает @channel_name в настройках
              ↓
Наш технический аккаунт (пул из 3–10 штук)
читает публичный канал через MTProto API (Telethon)
              ↓
Фильтрация и доставка результатов по watchlist каждого юзера
```

Именно так работают TGStat, Telemetr и все крупные Telegram-аналитики.

**Ограничения:**
- Только публичные каналы (`@username`) — приватные недоступны
- Один технический аккаунт комфортно читает ~500–1000 каналов
- При росте — просто добавляем новый аккаунт в пул

**Преимущества:**
- Никаких пользовательских сессий на сервере
- Нет юридических рисков
- Каналы дедуплицируются между пользователями (если двое следят за одним каналом — читаем один раз)

---

## 3. User Journey

1. **Регистрация** — email + пароль или OAuth (Google)
2. **Создание watchlist** — добавить публичные каналы (`@channel_name`), выбрать топик
3. **Настройка алертов** — порог score, минимум каналов, язык уведомлений
4. **Настройка доставки** — Telegram Bot (свой токен) и/или Webhook URL
5. **Получение сигналов** — алерты в реальном времени через выбранный канал

### Компоненты

```
┌─────────────────────────────────────────────────────┐
│                    FastAPI                          │
│   Auth · Watchlist CRUD · Delivery config           │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│                 Celery Beat                         │
│         Планировщик задач — каждую минуту           │
│         enqueue batch_task per active user          │
└────────────────────┬────────────────────────────────┘
                     │
┌────────────────────▼────────────────────────────────┐
│              Redis — очереди задач                  │
│   batch:user_a · batch:user_b · score:global        │
└───────┬──────────────────┬──────────────────────────┘
        │                  │
┌───────▼──────┐   ┌───────▼──────────────────────────┐
│  Batch       │   │  Scorer Worker (каждые 5 мин)     │
│  Workers     │   │  velocity score                   │
│  (per user)  │   │  cross-channel merge              │
│              │   │  topic filter per user            │
│  dedup       │   │  alert if score > threshold       │
│  normalize   │   └───────────────────────────────────┘
│  embed       │
│  cluster     │
└───────┬──────┘
        │
┌───────▼──────────────────────────────────────────────┐
│           PostgreSQL + pgvector                      │
│  users · watchlists · clusters · scores · alerts     │
└──────────────────────────────────────────────────────┘
```

### Технический стек

| Слой | Технология | Назначение |
|---|---|---|
| API | FastAPI | REST API, auth, webhook |
| Парсинг | Telethon (MTProto) | Чтение публичных каналов |
| Очереди | Celery + Redis | Батч-обработка, планировщик |
| ML / NLP | sentence-transformers | Embeddings для кластеризации |
| Дедупликация | MinHash (datasketch) | Похожие тексты |
| Хэш фото | imagehash + Pillow | Похожие изображения |
| БД | PostgreSQL + pgvector | Хранение, векторный поиск |
| Кэш / очереди | Redis | Буфер постов, Celery broker |
| Доставка | Telegram Bot API · Webhook | Алерты пользователю (см. раздел ниже) |

### Pipeline (батч каждую минуту)

```python
async def run_batch(user_id: int):
    raw_posts = await drain_user_queue(user_id)   # из Redis
    if not raw_posts:
        return

    posts = dedup.run(raw_posts)                  # MinHash
    posts = await normalize.run(posts)            # перевод, очистка
    vectors = await embed.run(posts)              # sentence-BERT
    clusters = cluster.run(posts, vectors)        # cosine similarity

    await save_clusters(clusters, user_id)        # → Postgres
```

### Scorer (каждые 5 минут)

```
viral_score = (velocity × 0.4) + (engagement × 0.35) + (cross_channel × 0.25)

velocity      = Δchannel_count / Δhours   (log-scale)
engagement    = (views + forwards×3 + reactions×2) / channel_avg
cross_channel = unique_channels_count / watched_channels_count
```

### Доставка алертов

Пользователь выбирает один или оба канала доставки:

**Вариант 1 — Telegram Bot (пользователь даёт нам токен своего бота)**

```
Пользователь создаёт бота через @BotFather → получает токен
              ↓
Вставляет токен в настройки TrendPulse
              ↓
Наш notifier.py шлёт сообщение через этот токен в нужный chat_id
```

Плюс: алерт приходит от имени своего бота, пользователь полностью контролирует бота.

**Вариант 2 — Webhook (пользователь указывает URL)**

```
Пользователь указывает endpoint: https://his-service.com/webhook
              ↓
При триггере alert — POST запрос с JSON payload
```

```json
{
  "event": "viral_alert",
  "topic": "crypto",
  "title": "Bitcoin ETF approval",
  "score": 94,
  "channels_count": 47,
  "first_seen": "2025-06-08T14:02:00Z",
  "velocity": 2.3
}
```

Плюс: интеграция с любым сервисом — Slack, Discord, собственный сервис, n8n, и т.д.

Оба варианта можно активировать одновременно.

## 5. Мульти-тенантность

Каждый пользователь полностью изолирован:

- Посты, кластеры, скоры в Postgres имеют `user_id`
- Celery очереди именованы per-user: `batch:user_{id}`
- `max_instances=1` — батч одного юзера не запускается параллельно
- Каналы дедуплицируются на уровне технических аккаунтов (один канал читается один раз, даже если его добавили 100 юзеров)

## 6. Монетизация

### Оплата — криптовалюта

Никакого Stripe. Принимаем крипту напрямую:

| Сеть | Токен | Почему |
|---|---|---|
| Solana | USDC / SOL | Быстро, дёшево, целевая аудитория уже там |
| Ethereum | USDC / ETH | Широкое распространение |
| TON | USDT / TON | Нативно для Telegram-аудитории |

**Механика:** пользователь оплачивает подписку переводом на наш кошелёк → транзакция верифицируется автоматически → план активируется. Для автоматизации — использовать `NOWPayments` или `CoinGate` API (они поддерживают webhook на успешную оплату).

## 7. Compliance и ограничения

### Telegram

| Разрешено | Запрещено |
|---|---|
| Чтение публичных каналов через MTProto | Хранение session_string пользователя |
| Пул технических аккаунтов для мониторинга | Автоматизированный спам и флуд |
| Telegram Bot API для уведомлений | Чтение приватных каналов без разрешения |
| TGStat / Telemetr API как легальная альтернатива | Продажа сырого контента постов |

**Rate limits:** при FLOOD_WAIT ошибках — exponential backoff, ротация между аккаунтами пула.

### GDPR / Персональные данные

- Сырой контент постов не хранить дольше 48 часов (только метрики и векторы)
- Пользователь может удалить аккаунт и все данные одним запросом
- Privacy policy обязательна с явным описанием обработки данных
- Логировать только агрегированные метрики, не содержимое сообщений

### Юридические риски

- **Низкий:** чтение публичных каналов — публичная информация
- **Средний:** агрегация данных может попасть под "processing of public data" в ряде юрисдикций
- **Рекомендация:** зарегистрировать как SaaS-инструмент, не как "data broker"

## 9. Roadmap

Фазы 1–3 исходного плана (backend core, landing+frontend, монетизация) — **выполнены**
(эпики A–D, task-001…028; история — в [tasks-index](../tasks/tasks-index.md)).
Из фазы 4 готов API access (task-028); Twitter/X спроектирован (task-031, триггер $2k MRR).

Актуальный roadmap: [`../architecture/roadmap.md`](../architecture/roadmap.md).
Текущая волна — **Epic E «Path to revenue»** ([полные планы](./epics/README.md)): E0 ops-фундамент →
E1 первая польза за 30 секунд → E2 качество сигнала → E3 витрина-канал → E4 деньги без трения →
E5 цены/упаковка → E6 бизнес-метрики → E7 масштаб → E8 новые рынки. Цель: первый платящий → $2k MRR.

> Важный сдвиг против исходного плана: white-label и разовые отчёты **вычеркнуты из ближайших
> 6 месяцев** (расфокус, см. [E5](./epics/epic-e5-pricing-packaging.md)); Free-план переосмысляется
> как воронка (готовые наборы + задержка алертов вместо «5 своих каналов real-time»); цены
> пересматриваются от ценности (скорость/точность/API), а не от числа каналов.
