# E6 · Бизнес-метрики (видеть, что работает)

**Owner:** backend + product · **Волна:** после E4

## Зачем

Техническая observability есть (Sentry, trace-id — task-024), бизнес-метрик нет: не видно, кто доходит
до первого алерта, кто конвертится, кто и где отваливается. Задержка сигнала p50/p95 — **тот самый KPI,
который продаём** — закрывается раньше, в TASK-036 (E0).

## Задачи (defined)

| Task | Что | DoD кратко |
|------|-----|------------|
| TASK-050 · Воронка активации + конверсия/отток | События: регистрация → пак подключён → первый алерт доставлен → первая оценка 👍/👎 (данные TASK-042); конверсия Free→Paid, churn по дате истечения. Реализация: `log_event`-события + ежедневный Beat-агрегат в таблицу `business_metrics_daily` | видно, на каком шаге теряются люди; конверсия и отток считаются ежедневно |
| TASK-051 · Дашборд «деньги» | MRR, активные подписки по планам, средний чек, retention по когортам — из `subscriptions`/`billing_payments`; отдача: ops-эндпоинт/JSON + простая страница (или Grafana поверх Postgres — решить при планировании) | один экран: сколько зарабатываем, где теряем |

Зависимости: TASK-050 использует фидбек из TASK-042; обе после включения оплаты (E4/E5), иначе считать нечего.

## Финальная точка эпика

На одном экране: MRR/подписки/чек/retention, воронка активации с точками потерь, задержка сигнала
p50/p95 (из TASK-036). Решения по продукту принимаются по этим цифрам, не по ощущениям.
