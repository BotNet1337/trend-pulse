# E2 · Качество сигнала = удержание

**Owner:** backend + product · **Волна:** СЕЙЧАС

## Зачем

Из 10 алертов 8 мусорных → юзер уходит на второй неделе. Две конкретные проблемы в коде:
(1) `engagement = метрика / channel_avg`, где `channel_avg` — среднее по **текущей пачке**, а не
исторический фон канала (learnings task-008) — «спайк» не отличается от обычного дня;
(2) обратной связи нет вообще — система не знает, полезны ли её сигналы.

## Задачи (defined — task-док через `trendpulse-plan` при взятии)

| Task | Что | DoD кратко |
|------|-----|------------|
| TASK-041 · Historical engagement baseline | `channel_avg` считается по скользящему историческому окну канала (например, 7 дней, константа), а не по батчу. Touch: `scorer/` engagement-расчёт + хранение/выборка фона (agg по `posts`) | алерт срабатывает на отклонении от нормы канала; unit-тест: ровный канал не триггерит |
| TASK-042 · Фидбек 👍/👎 + precision per user | Inline-кнопки в Telegram-алерте (callback на наш бот-endpoint), `alert_feedback` (alert_id, verdict), метрика precision per user в observability. Touch: `alerts/formatting.py`, `alerts/backends.py` (reply_markup), новый API callback-роут, миграция | юзер оценивает в один тап; `log_event("alert_precision", …)` per user |
| TASK-043 · Адаптивный порог + анти-fatigue | Автоподстройка `Watchlist.threshold` по доле 👎 (медленно, с floor/ceiling-константами); лимит N алертов/час на юзера + группировка похожих. Touch: scorer trigger-path + dispatch-rate-guard | часто 👎 → порог растёт; не больше N алертов/час (константа) |

Порядок: 041 → 042 → 043 (043 питается данными 042).

## Финальная точка эпика

Видна precision-метрика по каждому юзеру, оценка алерта — один тап, порог подстраивается под реакцию,
alert fatigue ограничен. Целевой ориентир: ≥70% алертов с оценкой получают 👍 у активных юзеров.
