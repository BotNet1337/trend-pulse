## 2026-06-08 — TASK-001 Dev + infra environment
- **Lesson:** В Docker `COPY --from=<image:${TAG}>` НЕ поддерживает интерполяцию переменных. **Why:** buildkit резолвит `--from` до раскрытия ARG в строке образа → ошибка «variable expansion is not supported for --from». **How to apply:** выноси версионируемый внешний образ в отдельный stage (`FROM ghcr.io/astral-sh/uv:${UV_VERSION} AS uv`) и `COPY --from=uv …`. ARG до первого FROM = глобальный scope.
- **Lesson:** `uv sync --frozen` ставит и `default-groups` → dev-инструменты протекают в runtime-образ. **Why:** `[tool.uv] default-groups=["dev"]` удобен для локального CI, но образ один на api/worker/beat/migration_runner и уходит в prod. **How to apply:** в Dockerfile всегда `uv sync --frozen --no-dev`; dev-тулинг — только на хосте через `make ci`/`uv run`. Проверка: `docker run --rm <img> python -c "import pytest"` → ModuleNotFoundError.
- **Lesson:** Не клади пароль БД ни в код, ни в committed `deploy.env` (даже dev-дефолт задаёт паттерн). **Why:** CONVENTIONS — секреты только в `sensitive.env`/vault. **How to apply:** `Settings` собирает `database_url` из `POSTGRES_*` частей; пароль приходит из `sensitive.env`; `all.yml`/`deploy.env` держат только host/port/db/user.
- **Lesson:** `depends_on` без `condition: service_healthy` = гонка холодного старта (nginx → 502, celery → crash-loop до restart). **Why:** дефолт `service_started` ждёт только запуск процесса, не готовность. **How to apply:** дай api `healthcheck` (stdlib `urllib` на `/health`, без лишних пакетов), nginx → `depends_on api: service_healthy`; redis(healthy) в depends_on api/worker/beat.
- **Decision:** `ansible-unpack` на task-001 — dependency-free stub-рендерер `key: value` YAML → env (deploy.env из `all.yml`, sensitive.env из `vault.yml`, chmod 600). **Rationale:** локальный dev без `ansible` binary; реальный `ansible-vault` + secret-scan — **task-012**.
- **Gotcha (важно для task-002):** worker/beat сидят только на `postgres_net`/`redis_net` (`internal: true`) → **нет egress в интернет**. Telethon (MTProto) и pull моделей sentence-transformers в коллекторе сломаются. **How to apply:** в task-002+ дать worker egress-способную сеть (отдельная не-internal `egress`) ИЛИ кэшировать модель на build-time. Не наследовать молча.
- **Gotcha:** host-порт 443 часто занят другим локальным проектом (OrbStack/другой nginx). nginx.conf слушает только 80 локально (443 ssl — prod, под серты). **How to apply:** публикуй только `${HTTP_PORT:-80}:80`; 443 включай на prod вместе с TLS-блоком и сертами (ops/).
- **Gotcha (git hygiene):** `git rm --cached <dir>` оставляет файлы на диске, НО последующий `git checkout` на коммит/ветку, где dir был ещё tracked-then-removed, **удалит файлы с диска**. Так чуть не потеряли `.claude/` (хуки+скиллы) после merge finalize-PR. **How to apply:** untrack + gitignore в один заход и не переключайся на старые ревизии, ожидая файлы на диске; при потере — `git checkout <bootstrap-sha> -- <dir>` + `git restore --staged <dir>` (вернуть на диск, оставить untracked). Теперь `.claude`/`_bmad` untracked везде → переключения веток безопасны.

## 2026-06-08 — TASK-002 Data model (SQLAlchemy 2.0 · pgvector · multi-tenancy)
- **Lesson:** integration-тесты к БД нельзя гнать с хоста против compose-postgres (нет host-порта — изоляция из task-001). **Why:** network-design публикует только nginx. **How to apply:** для реального прогона integration-suite поднимай эфемерный `pgvector/pgvector:pg16` с проброшенным host-портом и `POSTGRES_HOST/PORT` env-override; продакшн-путь миграции проверяй отдельно через `migration_runner` + `docker exec psql` на изолированном стеке. (Оба пути прогнаны в G2.)
- **Decision:** baseline-миграция переиспользует СУЩЕСТВУЮЩИЙ `backend/migrations/` (task-001), не создаёт параллельный `alembic/`. `alembic.ini script_location=migrations`. **Rationale:** один alembic-скаффолд; план task-002 ошибочно говорил `alembic/` — диск важнее плана.
- **Decision:** pgvector без py.typed → mypy `ignore_missing_imports` через `[[tool.mypy.overrides]]` scoped на `pgvector.*` (НЕ inline `# type: ignore`). **Rationale:** CONVENTIONS запрещает inline-ignore; config-override локализован.
- **Gotcha:** размерность вектора (`EMBEDDING_DIM=384`) дублируется литералом в миграции (Alembic-миграции самодостаточны, не импортируют живой app-код) — ок, но **task-007 (pipeline) обязан матчить 384**, иначе Postgres отвергнет вставку. **How to apply:** при смене размерности — новая миграция + проверка `clusters.embedding` dim == EMBEDDING_DIM.
- **Gotcha:** репо-тесты строят схему через `Base.metadata.create_all`, а не миграцией → model↔migration drift не ловится round-trip-тестами (только `test_migrations` проверяет наличие таблиц). **How to apply:** при росте схемы добавить parity-check (autogenerate diff == empty).

## 2026-06-08 — TASK-003 Auth (fastapi-users + Google OAuth, JWT + cookie)
- **Lesson (КРИТИЧНО для локали):** `Secure` cookie НЕ отправляется браузером/curl по http. Локально стек http-only (:80) → и auth-cookie (`CookieTransport.cookie_secure`), и OAuth CSRF-cookie (`get_oauth_router(csrf_token_cookie_secure=...)`) должны быть `False` локально, иначе login/OAuth молча ломаются (401 / OAUTH_INVALID_STATE). **How to apply:** настройка `auth_cookie_secure` (код-дефолт True для prod, env `AUTH_COOKIE_SECURE=false` локально). Prod (task-012 `group_vars/prod.yml`) обязан выставить true.
- **Lesson:** fastapi-users 15 кладёт CSRF-токен в OAuth `state` (double-submit: state-JWT + cookie сверяются на callback). Ручной state → `OAUTH_INVALID_STATE`. **How to apply:** в тестах прогоняй реальный `/auth/google/authorize` чтобы получить валидный state + CSRF-cookie (мокая только `GoogleOAuth2.get_access_token`/`get_id_email`).
- **Lesson:** `oauth_accounts.expires_at` — INTEGER (int4, max 2147483647); мок unix-ts должен влезать (используй ~2000000000, не 9999999999).
- **Decision:** сохранили **int user id** (не UUID fastapi-users по умолчанию) — `FastAPIUsers[User,int]` + `IntegerIDMixin`, чтобы не ломать FK из task-002. Один `User` (расширили существующий `SQLAlchemyBaseUserTable[int]`), не второй. Миграция аддитивная (0002, down_revision 0001).
- **Decision (user-directive):** плоская раскладка `backend/src/` без уровня пакета `trendpulse/` — импорты top-level (`from config import`, `from storage...`, `from api...`). hatchling `[tool.hatch.build.targets.wheel] sources=["src"] only-include=["src"]`; mypy `mypy_path="src"` + `explicit_package_bases=true`; compose-команды `uvicorn api.main:app` / `celery -A celery_app`; alembic.ini `prepend_sys_path=src`. **Why:** запрошено пользователем; переопределяет CONVENTIONS §Repo layout (src-layout с пакетом). **How to apply:** новые backend-модули кладём прямо в `backend/src/<module>`, без `trendpulse.`-префикса.
- **Gotcha (prod-hardening долг → task-012):** (1) `auth_cookie_secure=true` на prod; (2) валидатор min-length ≥32 на `jwt_secret`/`oauth_state_secret` (дев-плейсхолдеры слабые, InsecureKeyLengthWarning); (3) шифрование Google токенов в `oauth_accounts` at rest; (4) `associate_by_email=True` полагается на Google-verified email (для строгости gate на is_verified). Защищённый контракт = `Depends(current_user)` (active, НЕ verified) — downstream-роуты выбирают gating осознанно.

## 2026-06-08 — TASK-004 Watchlist CRUD API
- **Decision (user-directive):** «одна junction-строка = один watchlist» — single channel на watchlist, адрес по числовому `id`; несколько каналов = несколько watchlists. Переопределяет multi-channel `channels: list` из task-doc. Лимит плана = макс. число watchlists на юзера (Free=5); `min_channels` = scoring-параметр, не счётчик каналов. **How to apply:** downstream (task-005 читает каналы watchlist'ов, task-006/008 берут topic+threshold) ожидают per-row watchlist с одним каналом.
- **Lesson:** sync-репозитории task-002 в async FastAPI-приложении — делай **sync `def` route-handlers** (FastAPI гонит их в threadpool; async-зависимость `current_user` всё равно резолвится) + sync session-dependency над `get_session`. Не тащить async-репозитории. (Auth остаётся на async-движке fastapi-users — два движка на одном psycopg-DSN, см. [[trendpulse-build-and-infra-gotchas]].)
- **Lesson (BOLA/IDOR):** tenant-изоляция держится на том, что `UserScopedRepository` НЕ наследует глобальный `get_by_id(by id)` — все выборки требуют `user_id`. Чужой/несущ. id → 404 (не 403/200, без утечки существования); tenant из токена (`get_tenant_user_id(user)`), НЕ из тела/пути; Pydantic `extra="forbid"` (нет mass-assignment `id`/`user_id`).
- **Gotcha:** дубликат `(user_id, channel_id, topic)` (unique из task-002) лови как IntegrityError → rollback → доменная ошибка → **409**, не 500. Покрой тестом (review поймал отсутствие).
- **Gotcha (seam-долг):** лимит — read-then-write (TOCTOU, неатомарен) + `len(list())` для count → task-010 (атомарный счётчик/индекс, пагинация). `validate_ref` сейчас format-only; когда task-005 даст `collector.registry` — проверь его на SSRF/timeout (вызывается на attacker-controlled handle на границе).

## 2026-06-08 — TASK-005 Source abstraction + Telegram collector
- **Lesson (session-генерация):** коды входа исчерпываются (`SendCodeUnavailableError`) после нескольких попыток; нестабильный MTProto-хэндшейк (`invalid new nonce hash` / `Server closed connection`) мешает доставке. Надёжный путь — **QR-login** (`client.qr_login()`, `exportLoginToken` — другой механизм, не коды): `development/scripts/get-telegram-session.sh` (обёртка: читает creds из sensitive.env, `uv run --with qrcode`, рисует QR в терминале). Сканировать может только владелец телефона — это неустранимый human-step (как код). См. `development/scripts/README.md`.
- **Lesson (egress):** worker сидел только в `internal: true` сетях → нет интернета (предсказано в task-001). Добавил **не-internal сеть `egress` только для worker** (collector→MTProto); api/postgres/redis остаются изолированы. Топология: worker = postgres_net+redis_net+egress.
- **Lesson (Telethon lifecycle — HIGH, поймал review):** клиент `connect()` на acquire без `disconnect()` = утечка сокетов/сессий в долгоживущем worker (+ `coroutine ignored GeneratorExit` в тестах). **Всегда** давай пулу/коллектору `aclose()` (дисконнект всех) + async-context-manager; в тестах/worker оборачивай в `async with`.
- **Decision:** ADR-001 порт реализован с разумными отклонениями от иллюстративной модели: `author: str | None`, `since: datetime | None` (backfill), `fetched_at` опущен. telethon — lazy-import (base.py SDK-free), mypy-override scoped на telethon.
- **Decision (user):** `POOL_MIN=1` на старте (одна dev-сессия), цель 3–10 — вернуть когда пул заполнен. FLOOD_WAIT mid-iteration может пере-выдать уже отданные посты → поглощается dedup по `external_id`.
- **Gotcha (secret hygiene):** StringSession = доступ к аккаунту уровня пароля; держать ТОЛЬКО в sensitive.env (gitignored)/vault, никогда в логах/repr/git/чате. `api_hash`, засвеченный в чате, стоит ротировать на my.telegram.org.

## 2026-06-08 — TASK-006 Celery infra (queues, beat, locks)
- **Decision (ADR-002 §2 refinement):** динамические per-user очереди `batch:user_{id}` **операционно непотребляемы** статическим worker `-Q` (Celery не умеет wildcard-подписку) → задачи копятся неисполненными. Перешли на **единую очередь `batch`** + per-user Redis-lock как гарант `max_instances=1`. Worker: `-Q celery,batch,score:global`. **Why:** lock уже даёт сериализацию по юзеру; шаринг очереди + горизонтальное масштабирование воркеров. **How to apply:** не плодить динамические очереди на тенанта; изоляция — через lock, не через имя очереди.
- **Lesson (review поймал HIGH):** проверяй, что worker реально ПОТРЕБЛЯЕТ очереди, в которые beat/диспетчер шлёт (не только default). G2 должна гонять beat→dispatch→consume, а не только `.delay()` в default.
- **Lesson (MEDIUM):** не зови `get_settings()` на уровне модуля — Settings с required-секретами (jwt_secret и т.п.) кинет ValidationError при простом импорте без env. Резолюй конфиг **лениво** внутри функции (`ttl: int | None = None` → settings при None).
- **Gotcha:** owner-checked Redis-lock release — атомарный CAS через `Redis.transaction` (WATCH/MULTI), не Lua → работает и на fakeredis (нет lupa). Сравнение токена идёт в bytes (`decode_responses=False`).
- **Gotcha:** postgres-том `development_postgres_data` переживает `make down`/`up` → данные прошлых задач (напр. watchlists из task-004) видны; полезно для G2 (диспетчер нашёл реального активного юзера), но помни при «чистых» прогонах (`docker volume rm`).

## 2026-06-09 — TASK-007 Batch pipeline (dedup→normalize→embed→cluster)
- **Lesson (HIGH, review поймал):** общий образ собирается `uv sync --frozen --no-dev` → `ml`-группа (torch/sentence-transformers) НЕ попадает в образ → worker падал бы ImportError на embed. **FIX:** Dockerfile build-ARG `INSTALL_ML`; **отдельный ml-worker-образ** (`APP_IMAGE_ML`, worker.yml `INSTALL_ML=true`), api/beat остаются на лёгком `APP_IMAGE`. **How to apply:** тяжёлые ML-деп — только в worker-образ через build-arg, не в общий; верь, что G2 «через worker» гоняет именно нужный образ.
- **Lesson:** `pytest.importorskip(...)` на уровне МОДУЛЯ исполняется на стадии collection даже для deselected (`-m 'not integration'`) тестов → при установленной опц. группе импортирует тяжёлый пакет и загрязняет `sys.modules`, ломая lazy-import-юнит-тесты. **How to apply:** skip-guard + heavy import — ВНУТРЬ тестовой функции, не на уровне модуля.
- **Lesson:** Redis read-and-clear (`lrange`+`delete`) неатомарен → конкурентный `rpush` между ними теряется. Делай за одну MULTI/EXEC-транзакцию (`pipe.lrange; pipe.delete; pipe.execute()`).
- **Decision:** embed — lazy singleton (импорт sentence_transformers ВНУТРИ `_load_model`, не на уровне модуля), имя модели из settings, выход строго `EMBEDDING_DIM`(384) с fail-fast. Шаги pure/immutable (frozen dataclasses, новые списки), платформо-независимы (не импортируют collector.telegram).

## 2026-06-09 — TASK-008 Scorer (viral score + alert trigger)
- **Decision:** виральный score — чистая детерминированная функция (overview §4: velocity·0.4+engagement·0.35+cross_channel·0.25; velocity=log1p(Δchannels)/Δhours; веса/факторы 3,2 — именованные; guard на zero-div). Платформо-независима (no collector import) — AC2-тест сканирует импорты score+tasks.
- **Lesson (идемпотентность alert):** unique `alerts(user_id, cluster_id)` (миграция 0003) + вставка под `session.begin_nested()` (SAVEPOINT) с catch IntegrityError → дубль откатывает ТОЛЬКО savepoint, не всю транзакцию (предыдущий Score/работа сохраняются). Гонка тиков безопасна.
- **Gotcha (footgun → task-009/011):** scoring идёт **per-TOPIC, не per-cluster** (нет post↔cluster FK в схеме task-002) → несколько свежих кластеров одной темы получают одинаковый score, каждый >порога = свой alert. **task-009 (delivery) обязан rate-limit'ить**; точный per-cluster score требует FK post→cluster (future). `channel_avg` сейчас = среднее окна (не исторический baseline) → engagement не «спайк-vs-норма». `Score` пишется каждый тик без upsert → ретенция в task-011.

## 2026-06-09 — TASK-009 Alert delivery (Telegram Bot + webhook)
- **Lesson (КРИТИЧНО, SSRF DNS-rebinding TOCTOU — обе стадии review+security поймали как HIGH):** валидировать URL резолвом DNS, а потом слать `httpx.post(url)` (повторный независимый резолв при коннекте) = **байпас** через low-TTL DNS (public IP на валидации, private/`169.254.169.254` на коннекте). **FIX:** резолв ОДИН раз → валидация всех IP (is_global deny-list) → **пиннинг TCP-коннекта к провалидированному IP** (кастомный `httpx.HTTPTransport`), Host/SNI/cert по реальному hostname, `follow_redirects=False`. Регресс-тест с разными IP на validate vs connect. **How to apply:** для любого user-supplied URL — resolve+validate+connect атомарны (pin IP), не доверяй повторному резолву httpx.
- **Lesson (mypy scope):** `[tool.mypy] packages=[...]` — при добавлении нового пакета его НАДО туда внести, иначе strict его НЕ проверяет (scorer/alerts были не покрыты — «mypy green» врал). После добавления всплыли реальные ошибки (Task generic args, `apply_async(args=tuple)` не list). **How to apply:** новый top-level пакет → сразу в mypy.packages; проверяй число «checked N source files».
- **Gotcha:** секрет в `@dataclass` → авто-`__repr__` его светит. `field(repr=False)` для токенов. bot_token/webhook_url plaintext at-rest → шифрование в task-012 (как OAuth-токены).
- **Decision:** delivery-config (telegram_bot_token/chat_id/webhook_url/plan) живёт на User (аддитивная миграция 0004); plan default "free" — seam, реальный биллинг task-010. dispatch_alert на default-очереди (worker уже потребляет), enqueue из scorer только для новых alert'ов.

## 2026-06-09 — TASK-010 Billing (NOWPayments crypto + plan enforcement)
- **Lesson (HIGH, idempotency vs multi-IPN):** платёжные шлюзы (NOWPayments) шлют НЕСКОЛЬКО webhook'ов на ОДИН платёж с одним id (waiting→confirming→confirmed→finished). Ключ идемпотентности должен быть «уже АКТИВИРОВАН» (terminal status), НЕ «видели payment_id» — иначе промежуточный IPN глушит активирующий и план не активируется никогда. `processed_at`/terminal-флаг ставь ТОЛЬКО на активации. Тестируй ПОСЛЕДОВАТЕЛЬНОСТЬ статусов, не один finished.
- **Lesson (webhook HMAC):** верифицируй HMAC `hmac.compare_digest` (constant-time) ДО парсинга/доверия телу; невалид/отсутствие/пустой секрет → reject (unset secret → 503, не верифицирует произвольное). NOWPayments подписывает sorted-key JSON — для надёжности лучше верифицировать по RAW-байтам (re-canonicalize рискует разойтись: ensure_ascii/float repr) — нужен реальный IPN-сэмпл для byte-проверки (долг).
- **Decision:** единая точка enforcement `assert_within_limit(session, user, resource)` (ADR-003); план писать ТОЛЬКО верифицированным IPN (no client-trust); invoice↔user binding + amount/currency cross-check против replay/cross-account; expired sub → effective FREE. PLAN_LIMITS — именованная таблица (overview §6), не литералы.
- **Lesson (test isolation, растущий suite):** по мере накопления integration-тестов всплывают: (a) тесты, требующие Redis/брокер (celery, scorer-enqueue) — скип-гардь Redis или делай side-effect устойчивым; (b) `[tool.mypy] packages` НЕ автоматический — каждый новый пакет добавляй вручную (alerts/scorer были не проверены strict). Гоняй ПОЛНЫЙ `-m integration` в G2, не только тесты текущей задачи.

## 2026-06-09 — TASK-012 Ops/IaC (Terraform + Ansible + secrets)

- **Lesson (4 витрины цены — grep-гейт обязателен):** цены жили в plans.py, frontend constants, landing (config + faq + features + seo!) и overview §6 — grep по старым значениям нашёл хвосты вне заявленного скоупа (seo.ts, faq.tsx). **How to apply:** при смене публичных значений — grep по старому значению ПО ВСЕМУ репо включая копирайт/SEO, а не только по «известным» файлам.
- **Gotcha (фикстуры paid-планов):** effective_plan откатывает план без активной Subscription — интеграционные фикстуры «pro-юзера» обязаны сидировать Subscription, иначе тесты тихо тестируют Free.
