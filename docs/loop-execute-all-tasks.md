# Runbook — выполнить ВСЕ задачи через `/loop`

Автономный прогон всех `docs/tasks/task-*.md` до `status: done`, каждая — отдельным PR, влитым в `main` после подтверждения.

## Как запускать

1. Запускай Claude Code **из** `apps/trendPulse` (там активны trendpulse-скиллы и хуки).
2. Предусловие: задачи в статусе `planned` (так и есть). Для PR нужен git-репозиторий + remote — луп забутстрапит (см. ниже), но `gh auth login` должен быть выполнен заранее.
3. Введи `/loop` **без интервала** (self-paced) и вставь промпт ниже. Луп остановится на каждом PR и спросит подтверждение на merge; ответь «подтверждаю» — продолжит со следующей задачей. Когда все `done` — завершится сам.

```
/loop
```

## Промпт (вставить после /loop)

> Ты — оркестратор автономного выполнения ВСЕХ задач TrendPulse. Рабочая директория: `/Users/macbookpro16/work/botnet/apps/trendPulse`. Источник истины — frontmatter `status` и блок `Checkpoints` (`current_step`) в `docs/tasks/task-*.md` + `docs/tasks/tasks-index.md`. Общайся по-русски. Соблюдай `docs/CONVENTIONS.md` и архитектуру в `docs/architecture/`.
>
> **ЦЕЛЬ:** довести каждую `docs/tasks/task-*.md` до `status: done` — каждую отдельным PR, влитым в `main` после подтверждения пользователя. Луп идемпотентен: каждую итерацию заново читай статусы с диска и продолжай с того места. Обрабатывай **ровно одну задачу за итерацию**.
>
> **BOOTSTRAP (один раз, если ещё не сделано):**
> - Если `apps/trendPulse` не git-репозиторий → `git init`, ветка `main`, начальный commit каркаса.
> - Проверь remote (`git remote -v` / `gh repo view`). Если репозитория/remote нет — **СПРОСИ** пользователя один раз: создать приватный через `gh repo create`? Не запускай стадию ship, пока remote не готов.
>
> **ИТЕРАЦИЯ:**
> 1. **Выбор задачи.** Прочитай все `task-*.md` (frontmatter `status`) и `tasks-index.md` (колонка Deps). Возьми задачу с наименьшим номером, у которой `status != done` И все зависимости уже `done`. Если actionable-задач нет, но остались незавершённые с невыполненными зависимостями → сообщи о блокировке и HALT. Если все `done` → переходи к ЗАВЕРШЕНИЮ.
> 2. **Статус «до» → «во время».** В выбранном task-доке: `status: in-progress`, `updated:` = сегодня, в Checkpoints выставь `lock` = id этого рана. Сохрани. (Жизненный цикл: `planned → in-progress → review → done`.)
> 3. **Выполнение.** Запусти скилл **`trendpulse-executor`** для этой `TASK-NNN`. Он войдёт на `current_step` и проведёт стадии **do (TDD RED→GREEN) → verify (G2, реальная behavioral-проверка) → review (adversarial, другой моделью) → security (5.5, если применимо по заметке в Checkpoints) → ship**, тикая Checkpoints/`current_step` и дописывая `## Details`. Все действия — через `make` (Makefile), версии из `version.env`, секреты из `sensitive.env`, сети по `network-design`.
>    - Блокирующие находки verify/review/security → executor уходит в `debug` (макс 2 цикла) и переспроверяет. Если не решилось за 2 цикла, нужен новый dependency/конфиг/секрет или продуктовое решение → **HALT и спроси пользователя**, не угадывай.
> 4. **Валидация (перед ship).** Убедись: все Acceptance Criteria (Given/When/Then) выполнены с доказательствами; Checkpoints 1–5 (+5.5 если применимо) = `[x]`; diff строго в рамках Scope; `make ci` зелёный (ruff + mypy + pytest); реальная G2-проверка пройдена.
> 5. **PR (одна логика = один PR).** Стадия ship: ветка `gsd/phase-{NNN}-{slug}`, Conventional Commit (код + доки в одном PR — репозиторий один), `git push -u`, `gh pr create` с описанием: summary, verify-evidence (вкл. реальную behavioral-проверку), чеклист AC, ссылка на task-док. В task-доке: `status: review`, запиши `branch` и номер PR во frontmatter/Checkpoints. **Никогда не коммить/мёржи в `main` напрямую.**
> 6. **Статус «после» + подтверждение.** Покажи пользователю: что сделано, ссылку на PR, verify-доказательства, чеклист AC. **СПРОСИ подтверждение на merge в `main`.** Без явного подтверждения — НЕ мёржи; заверши ход и жди ответа (не планируй wakeup только ради ожидания).
> 7. **Merge (после «подтверждаю»).** `gh pr merge --squash --delete-branch`. Затем: `status: done`, тикни Checkpoints 6 ship + 7 learnings, `current_step: done`, очисти `lock`. Обнови `tasks-index.md` (статус → done). Выполни стадию **learnings** (допиши `docs/learnings.md`; долгоживущие решения → ADR/memory).
> 8. **Дальше.** Краткий итог и переход к следующей итерации (следующая actionable-задача).
>
> **ПРАВИЛА:** одна задача за итерацию; уважай зависимости (task-001 первой); только PR-flow (хук bash-guard блокирует commit/push в main); всё через `make`, не raw `docker compose`; любая неоднозначность/нехватка решения → HALT и вопрос, а не догадка.
>
> **ЗАВЕРШЕНИЕ:** когда все `task-*.md` имеют `status: done` — выведи финальную сводку (таблица: задача · PR · итог) и **заверши луп** (не планируй следующий wakeup).
